# Assignment 3 Logbook: Decoded WAL and Stress Logs
**Team Dragons | CS432 Database Systems | Assignment 3**

This document explains the `logs/` folder in a decoded, instructor-friendly format.
All logs are append-only JSON-line files generated by the Assignment 3 transaction
engine (`module_a/engine/wal.py`) and the Module B stress-testing suite.

The WAL is the **sole durability channel** for the engine. Because the B+ Trees are
entirely in-memory, a process restart would lose all state unless the WAL replays
the committed transactions. Every log file here is direct evidence of that mechanism.

## 1. What These Logs Represent

Each line in a `.log` file is one WAL record flushed to disk via `os.fsync()`
before the writing method returns. Record types:

| Type | When Written | Meaning |
|------|-------------|--------|
| `SCHEMA` | `create_table()` call | Recreates the relation definition on recovery |
| `BEGIN` | `begin()` | Opens a transaction; acquires the serialized lock |
| `OP` | `tx.insert / tx.update / tx.delete` | Stages a mutation **before** any B+ Tree is touched (write-ahead property) |
| `PREPARE` | Inside `commit()` before atomic apply | Validation fence: schema + FK checks passed |
| `COMMIT` | After all B+ Tree writes succeed | Durable marker; `os.fsync()` called here |
| `ROLLBACK` | `rollback()` or exception in `commit()` | Abort; includes `reason=commit_failed` when triggered by validation or apply failure |

**Write-ahead guarantee:** `OP` records are written to the WAL before
`_apply_operations_atomically` is called. A crash between `OP` and `COMMIT` leaves
no `COMMIT` record, so recovery discards the transaction. This is what makes the
system ACID-Atomicity- and ACID-Durability-correct.

Canonical source files:

- WAL writer: `module_a/engine/wal.py`
- Transaction + recovery manager: `module_a/engine/transactional_db.py`

## 2. WAL Record Format (Decoded)

Each record is a compact JSON object on a single line:

```json
{"type":"BEGIN",  "tx_id":"a3f7-...","ts":"2026-04-05T..."}
{"type":"OP",     "tx_id":"a3f7-...","op":"insert","table":"files","key":1001,"value":{"vault_id":10,"file_size":256,...},"ts":"..."}
{"type":"PREPARE","tx_id":"a3f7-...","ts":"..."}
{"type":"COMMIT", "tx_id":"a3f7-...","ts":"..."}
```

Field reference:

- `type`: lifecycle stage.
- `tx_id`: UUID grouping all records for one transaction.
- `op`: row action (`insert`, `update`, `delete`) -- present on `OP` records only.
- `table`: relation name (one of the seven GhostDrop domain tables).
- `key`: integer primary key.
- `value`: full row payload as a JSON object.
- `reason`: rollback cause (`commit_failed` = validation rejected or apply failed).
- `ts`: UTC ISO-8601 timestamp at record write time.

## 3. Recovery Rule (How Logs Are Replayed)

Recovery behavior in `transactional_db.py`:

1. Read all records in order.
2. Recreate tables from `SCHEMA` records.
3. Buffer operations after `BEGIN` by `tx_id`.
4. Drop buffered operations if `ROLLBACK` is seen.
5. Apply buffered operations only when `COMMIT` is seen.
6. Ignore transactions left incomplete (for example `BEGIN` with no `COMMIT`).

Implication:

- `BEGIN + OP` without `COMMIT` is intentionally non-durable.
- `PREPARE + COMMIT` means committed state survives restart.

## 5. How to Audit Any Single Transaction Quickly

Use this workflow:

1. Pick a `tx_id` from a `BEGIN` line.
2. Collect all lines with that `tx_id`.
3. Verify legal lifecycle:
   - Success path: `BEGIN -> OP... -> PREPARE -> COMMIT`
   - Failure path: `BEGIN -> OP... -> ROLLBACK`
4. Check no transaction has both `COMMIT` and `ROLLBACK`.
5. For crash scenarios, verify incomplete transactions have no `COMMIT`.

## 6. What Instructors Usually Look For in These Logs

- Schema replay evidence (`SCHEMA` lines with `index_type="bplustree"`).
- Proper WAL ordering before durability claims.
- Clear rollback records for invalid/failed transactions.
- No partial-commit traces in failure scenarios.
- Restart/recovery behavior aligned with `COMMIT` presence.

## 7. Practical Reading Notes

- Timestamps are UTC with offset.
- Logs are append-only; reruns can produce fresh files or appended histories depending on script reset behavior.
- Large logs (`module_b_stress.log`, `stress_graph.log`) should be analyzed by counts and transaction lifecycle patterns rather than line-by-line manually.

## 8. Quick Command Snippets for Review

```powershell
# List logs with size/time
Get-ChildItem .\logs | Select-Object Name,Length,LastWriteTime

# View first 40 lines of a log
Get-Content .\logs\acid_suite_consistency.log -TotalCount 40

# Count record types in one file (PowerShell quick view)
Get-Content .\logs\acid_suite_isolation.log |
  ForEach-Object { ($_ | ConvertFrom-Json).type } |
  Group-Object | Sort-Object Count -Descending
```

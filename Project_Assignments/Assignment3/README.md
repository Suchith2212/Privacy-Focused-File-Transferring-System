## Architecture

### Storage Layer

The B+ Tree from Assignment 2 is the **only** storage backend. Each of the seven
GhostDrop domain relations is stored in its own independent B+ Tree instance
(order 4, max 3 keys per node). No external storage engine or database is used.

```
vaults            --> BPlusTree     sessions          --> BPlusTree
inner_tokens      --> BPlusTree     download_logs     --> BPlusTree
files             --> BPlusTree     expiry_jobs       --> BPlusTree
portfolio_entries --> BPlusTree
```

### Transaction Layer

All transactional logic lives in `module_a/engine/transactional_db.py`.
The key class is `TransactionalDatabaseManager`.

```
TransactionalDatabaseManager
|
+-- begin()
|     Acquires a global threading.RLock.
|     Allocates a UUID transaction ID.
|     Writes a BEGIN record to the WAL.
|     Returns a Transaction object the caller uses to stage operations.
|
+-- tx.insert / tx.update / tx.delete
|     Appends a TxOperation to the Transaction's in-memory buffer.
|     Simultaneously writes an OP record to the WAL (durable immediately).
|     The B+ Trees are NOT touched yet.
|
+-- commit(tx)
|     Step 1 -- Validation:
|       _validate_transaction(ops) builds a projected snapshot of the database
|       as it would look after all staged operations. Runs _validate_row_schema
|       (field types, required fields, enum values) and _validate_cross_constraints
|       (FK checks across all 7 relations) against this snapshot. Any violation
|       raises ValueError and aborts before touching any B+ Tree.
|     Step 2 -- WAL PREPARE:
|       Writes a PREPARE record (pre-commit fence).
|     Step 3 -- Atomic apply:
|       _apply_operations_atomically walks the staged buffer. For each op it
|       saves the pre-image to an undo-stack before writing to the B+ Tree.
|       If any write fails, the undo-stack is replayed in reverse to restore
|       the original state, then the error is re-raised.
|     Step 4 -- WAL COMMIT + fsync:
|       Writes a COMMIT record. Calls os.fsync() to guarantee the COMMIT is
|       on disk before the method returns to the caller.
|     Step 5 -- Release lock.
|
+-- rollback(tx)
|     Writes a ROLLBACK record to the WAL.
|     Discards the staged operation buffer.
|     Releases the RLock.
|
+-- _recover_from_wal()  [called automatically on startup]
      Reads the WAL file sequentially:
        SCHEMA   -> recreate the relation if not already present
        BEGIN    -> start a pending transaction buffer
        OP       -> append to the buffer for that tx_id
        ROLLBACK -> discard that tx_id's buffer
        COMMIT   -> validate and replay the buffer into the B+ Trees
      Any tx_id still in the buffer at end-of-file was never committed
      and is silently discarded. Recovery never writes new WAL records.
```

### Write-Ahead Log (WAL)

Implemented in `module_a/engine/wal.py`. Each WAL entry is one JSON object
written with `os.fsync()` to guarantee durability:

```json
{"type": "SCHEMA",   "table": "vaults", "index_type": "bplustree", "order": 4, "ts": "..."}
{"type": "BEGIN",    "tx_id": "a3f7...", "ts": "..."}
{"type": "OP",       "tx_id": "a3f7...", "op": "insert", "table": "files", "key": 1001, "value": {...}, "ts": "..."}
{"type": "PREPARE",  "tx_id": "a3f7...", "ts": "..."}
{"type": "COMMIT",   "tx_id": "a3f7...", "ts": "..."}
{"type": "ROLLBACK", "tx_id": "a3f7...", "reason": "commit_failed", "ts": "..."}
```

The write-ahead property is guaranteed because OP records are written to the WAL
**before** `_apply_operations_atomically` is called. If the process crashes
between OP and COMMIT, recovery sees no COMMIT and discards the transaction.

### Isolation and Locking

The engine implements **Strict Two-Phase Locking (S2PL)** using a single
`threading.RLock`:

- The lock is **acquired** at `begin()` and held for the entire transaction.
- The lock is **released** only inside `commit()` or `rollback()`, after all
  WAL writes complete.
- Non-transactional reads go through `ReadOnlyTableView`, which acquires
  the same lock per read, ensuring readers always see committed state.
- This provides the **Serializable** isolation level: concurrent transactions
  are effectively serialized, preventing dirty reads, lost updates, and
  phantoms.

### Consistency Enforcement

Two validation layers run at every commit:

1. **Row-level schema** (`_validate_row_schema`): checks field types,
   required fields, and allowed enum values (e.g. `status` must be one of
   `ACTIVE`, `EXPIRED`, `DELETED`).

2. **Cross-table FK integrity** (`_validate_cross_constraints`): for every
   relation, verifies that all referenced PKs exist in the projected post-commit
   state. FK rules enforced:
   - `inner_tokens.vault_id` -> exists in `vaults`
   - `files.inner_token_id` -> exists in `inner_tokens` in the same vault
   - `files.vault_id` -> exists in `vaults`
   - `download_logs.file_id` -> exists in `files`
   - `download_logs.session_id` -> exists in `sessions`
   - `expiry_jobs.vault_id` -> exists in `vaults`
   - `portfolio_entries.vault_id` -> exists in `vaults`
   - `portfolio_entries.owner_token_id` and `created_by_token_id` -> exist
     in `inner_tokens` belonging to the same vault

Any violation raises `ValueError` and causes the commit to write
`ROLLBACK reason=commit_failed` without touching any B+ Tree.

## ACID Properties and Test Evidence

| Property | Engine Mechanism | Test Coverage |
|----------|-----------------|---------------|
| **Atomicity** | WAL staging + undo-stack on partial failure; crash-before-commit discarded by recovery | 3 test cases: crash, rollback, mid-commit injection |
| **Consistency** | Schema validation + FK cross-constraints at commit on projected state | 3 test cases: negative file size, missing vault ref, missing token ref |
| **Isolation** | Global `RLock` (Strict 2PL) + `ReadOnlyTableView` for committed-only reads | 3 test cases: concurrent inserts, no dirty read, write-conflict ordering |
| **Durability** | `os.fsync()` after every WAL COMMIT; full replay on startup | 4 test cases: single restart, 7-relation restart, incomplete tail, idempotent replay |

**Total Module A tests: 16/16 PASS**

### The 5 New Tests Added in This Submission

These go beyond the base rubric to prove harder correctness scenarios:

| Test | What It Proves |
|------|----------------|
| **Isolation (serialized write conflict ordering)** | Two sequential updates to the same row produce a deterministic final value matching T2's write, not T1's stale overwrite |
| **Atomicity (mid-commit failure injection)** | If a B+ Tree write fails after a partial apply, the undo-stack reverses all prior writes in the same batch |
| **Negative path: bypass is not durable** | Inserting directly into the raw B+ Tree produces a row visible now but absent after WAL recovery -- proves the WAL boundary is enforced |
| **Durability (multiple commits + incomplete tail)** | Three committed transactions all survive; a fourth that was staged but not committed is discarded |
| **Recovery (idempotent replay)** | Two separate recovery passes from the same WAL produce identical state, and the WAL file is unchanged after each pass |
